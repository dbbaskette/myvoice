# Format Add-On: Demo Script

Layer this on top of the master style guide for live demos (conference stage, customer call, internal review).

## Time budget

Aim to fill 80% of the allotted time speaking. The other 20% is buffer for things going wrong, a recovery, or one good audience question. A 20-minute slot is a 16-minute script.

## Structure

1. **Cold open:** "Let me show you what happens when…" Tension first. No terminal preambles.
2. **Setup beat (1 sentence):** What the audience will see in the next two minutes.
3. **Narrate-while-typing:** Every command spoken in plain English before you press enter. Intent first, syntax second.
4. **Named checkpoints:** "At this point you should see X." Lets the audience self-verify, lets you catch a misstep early.
5. **Recovery script per step:** For each command, write the one-line "if this fails, I do this" plan in the script margin. You won't read it on stage; you'll have rehearsed it.
6. **Stick the landing:** Close with the time savings or capability the demo proved. Not "any questions?". A concrete claim.

## Required elements

- A "what would normally take X" callout for every step that demonstrates a time savings (Principle B). The whole point.
- Every screen action narrated before the click.
- Backup plan for the network. Local copy of every external dependency.
- One named Marvel reference per demo if it lands naturally. Optional but on-brand.

## Forbidden in demo format

- "Let me just open my terminal."
- "Bear with me while this loads."
- More than three levels of menu navigation visible on screen at once.
- Reading commands aloud instead of narrating intent.
- Switching to slides mid-demo unless the slide is a single visual punch.
- Apologizing for the demo gods. If something breaks, work the recovery script and keep moving.
